Case study · Same agent, new framework

The same agent, redrawn as a state graph.

A companion rebuild of the privacy-assessment agent — this time on LangGraph instead of a hand-rolled loop — with an honest account of exactly what was verified before publishing.

// graph structure — langgraph_agent.py
START → agent
tool call→tools→agent(loop)
submit_privacy_assessment→finalize→END
no tool call→END

Why this exists

The original agent is implemented directly against the tool-use wire API on purpose — a stronger demonstration of understanding what a framework like LangGraph does underneath than importing it outright. This repo is the natural follow-up: the same agent, same tool contracts, same tests it can borrow — rebuilt on the actual framework, so both "understands the mechanics" and "fluent in the framework" are backed by real code.

What was verified, and how

This was originally built without PyPI access to install langgraph itself. Rather than quietly ship untested graph-wiring code, here's exactly how the gap was closed.

✓ Verified without the package — 8 tests
  • agent_node, should_continue, tools_node, finalize_node — plain functions, no LangGraph import
  • A full manual walkthrough composing all four in sequence, asserting the trace matches the hand-rolled agent exactly
  • The deterministic risk-level guarantee, re-verified on this third runner
✓ Verified on real LangGraph — 9/9 passing
  • build_graph() and run_agent() — confirmed on langgraph, pytest 8.1.2, Python 3.11.9
  • python langgraph_agent.py runs end-to-end, producing a trace and draft identical to the pre-verified manual walkthrough
  • Along the way, verification surfaced a real bug: an optional xgboost import only caught ImportError, not the AttributeError a real setuptools/xgboost version mismatch throws — fixed and confirmed

What verifying it actually found

The value of running this for real rather than trusting the manual walkthrough: it surfaced a genuine bug. attacker_model.py's optional XGBoost import only handled ImportError — on a real machine with a version mismatch between an older xgboost and newer setuptools, the import raised AttributeError instead, which crashed the whole module rather than triggering the scikit-learn fallback it was supposed to. "Optional dependency" fallback logic needs to anticipate more than just "not installed" — a lesson that only shows up by actually running code somewhere other than the environment it was written in.

What's reused, not reimplemented

TOOL_SCHEMAS

Same tool contracts

Imported from the original repo's agent.py rather than redefined a third time — one source of truth for what each tool does, regardless of which runner calls it.

dispatch_tool

Same execution logic

The functions that actually call Confluence, the vector store, and the attacker model are unchanged — this rebuild only changes how the control flow is expressed.

ScriptedMockAgentClient

Same offline demo behavior

The same scripted mock model drives both the hand-rolled agent and this graph — so their outputs are directly comparable, not just superficially similar.
